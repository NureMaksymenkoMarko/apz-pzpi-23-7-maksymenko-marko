3 ОПИС ВИКОНАНОЇ РОБОТИ
Патерн проєктування Strategy
    Патерни проєктування використовуються для того, щоб не створювати архітектурні рішення з нуля у кожній схожій ситуації. Вони описують перевірені способи організації класів та об’єктів, які допомагають зменшити складність коду, підвищити його гнучкість і зробити систему більш зручною для подальшого супроводу. Одним із таких рішень є поведінковий патерн Strategy.
    Патерн Strategy застосовується тоді, коли в системі існує декілька варіантів виконання однієї й тієї самої операції. Наприклад, програма може мати різні способи розрахунку вартості доставки, різні правила нарахування знижок, різні алгоритми сортування або кілька методів обробки даних. Замість того щоб записувати всі ці варіанти в одному великому методі через умовні оператори, Strategy пропонує винести кожен алгоритм в окремий клас.
    Головна ідея патерну полягає у тому, що об’єкт, який використовує алгоритм, не повинен знати його внутрішню реалізацію. Він працює лише зі спільним інтерфейсом стратегії. Завдяки цьому алгоритм можна замінити без зміни основної логіки програми.
Формальне визначення патерну
    Strategy — це поведінковий патерн проєктування, який визначає сімейство алгоритмів, інкапсулює кожен алгоритм в окремому класі та забезпечує їх взаємозамінність під час виконання програми. Клієнтський код при цьому залежить не від конкретної реалізації, а від абстрактного інтерфейсу.
    Іншими словами, патерн дозволяє відокремити логіку вибору поведінки від логіки її виконання. Якщо у програмі з’являється новий алгоритм, розробник додає новий клас стратегії, а не переписує вже існуючий код. Це відповідає принципу відкритості/закритості: програмні сутності мають бути відкритими для розширення, але закритими для модифікації.
Необхідність використання патерну Strategy
    Потреба у патерні Strategy найчастіше виникає тоді, коли один клас починає відповідати за занадто велику кількість варіантів поведінки. На перших етапах розробки така реалізація може виглядати простою: достатньо додати кілька операторів if або switch. Проте з часом кількість умов збільшується, код стає довшим, а кожна зміна створює ризик помилок у вже працюючій частині програми.
    Наприклад, якщо сервіс замовлень підтримує стандартну доставку, експрес-доставку та самовивіз, розробник може реалізувати розрахунок ціни в одному методі. Але якщо пізніше потрібно додати міжнародну доставку, доставку кур’єром або акційний тариф, цей метод доведеться постійно редагувати. Це ускладнює тестування і робить код менш зрозумілим.
    Strategy розв’язує цю проблему через розподіл відповідальності. Клас Context відповідає за використання алгоритму, але не містить його деталей. Конкретні класи Strategy відповідають тільки за один варіант поведінки. У результаті кожну стратегію можна окремо змінювати, тестувати або замінювати.
Структура та ролі учасників патерну
    Архітектура патерну Strategy складається з кількох основних учасників. Кожен з них має окрему роль і забезпечує гнучку взаємодію між клієнтським кодом та алгоритмами.
    Strategy — це інтерфейс або абстрактний клас, який описує загальний метод для всіх алгоритмів. Він не містить конкретної бізнес-логіки, а лише задає контракт, як саме Context буде звертатися до стратегії.
    ConcreteStrategy — це конкретна реалізація стратегії. Кожен такий клас реалізує один варіант алгоритму. Наприклад, можуть існувати класи StandardDeliveryStrategy, ExpressDeliveryStrategy та PickupStrategy, які мають однаковий метод розрахунку, але повертають різні результати.
    Context — це клас, який використовує стратегію. Він зберігає посилання на об’єкт типу Strategy і викликає його метод тоді, коли потрібно виконати певну дію. Context не повинен знати, який саме клас реалізує алгоритм.
    Client — це частина програми, яка створює потрібну стратегію і передає її у Context. Саме клієнтський код зазвичай визначає, який варіант поведінки потрібно застосувати в конкретній ситуації.
Порівняння прямої реалізації та використання Strategy
    У прямій реалізації всі варіанти алгоритму часто зберігаються в одному методі. Такий підхід простий для невеликої кількості варіантів, але він швидко втрачає зручність, коли бізнес-правила змінюються або розширюються. Один клас починає містити багато умовних операторів, а його логіка стає перевантаженою.
    При використанні Strategy код організовується інакше. Замість великого методу з умовами створюється набір окремих класів, кожен з яких відповідає за конкретний алгоритм. Context працює через інтерфейс і не залежить від деталей реалізації. Завдяки цьому можна змінити алгоритм у процесі виконання програми або додати новий алгоритм без редагування основного класу.
    Отже, прямий підхід може бути доречним для дуже простих задач, але Strategy краще підходить для систем, де поведінка має змінюватися, розширюватися або обиратися динамічно.
Приклад програмної реалізації
    У межах практичної роботи було розглянуто приклад сервісу розрахунку вартості доставки. У першому варіанті реалізації метод CalculateDeliveryPrice() містить умовні оператори, які перевіряють тип доставки. Такий код прив’язує всі алгоритми до одного класу і змушує змінювати цей клас при появі нового способу доставки.
public class DeliveryService
{
    public decimal CalculateDeliveryPrice(string deliveryType)
    {
        if (deliveryType == "standard")
            return 80;

        if (deliveryType == "express")
            return 160;

        if (deliveryType == "pickup")
            return 0;

        throw new ArgumentException("Unknown delivery type");
    }
}

    У другому варіанті використовується патерн Strategy. Спочатку створюється інтерфейс IDeliveryStrategy, який визначає метод Calculate(). Потім для кожного способу доставки створюється окремий клас, що реалізує цей інтерфейс.
public interface IDeliveryStrategy
{
    decimal Calculate();
}

public class StandardDeliveryStrategy : IDeliveryStrategy
{
    public decimal Calculate() => 80;
}

public class ExpressDeliveryStrategy : IDeliveryStrategy
{
    public decimal Calculate() => 160;
}

public class PickupStrategy : IDeliveryStrategy
{
    public decimal Calculate() => 0;
}

    Після цього створюється клас OrderContext, який приймає стратегію і використовує її для розрахунку вартості доставки. Context не перевіряє тип доставки і не містить умовних операторів. Він лише викликає метод стратегії.
public class OrderContext
{
    private IDeliveryStrategy deliveryStrategy;

    public OrderContext(IDeliveryStrategy deliveryStrategy)
    {
        this.deliveryStrategy = deliveryStrategy;
    }

    public void SetDeliveryStrategy(IDeliveryStrategy deliveryStrategy)
    {
        this.deliveryStrategy = deliveryStrategy;
    }

    public decimal GetDeliveryPrice()
    {
        return deliveryStrategy.Calculate();
    }
}

    Клієнтський код може створити потрібну стратегію і передати її в контекст. Якщо користувач змінює спосіб доставки, програма може замінити стратегію без зміни самого класу OrderContext.
var order = new OrderContext(new StandardDeliveryStrategy());
Console.WriteLine(order.GetDeliveryPrice()); // 80

order.SetDeliveryStrategy(new ExpressDeliveryStrategy());
Console.WriteLine(order.GetDeliveryPrice()); // 160

Послідовність взаємодії об’єктів
    Робота патерну Strategy відбувається у чіткій послідовності. Спочатку клієнтський код визначає, який алгоритм потрібен у конкретній ситуації. Потім створюється об’єкт конкретної стратегії, наприклад ExpressDeliveryStrategy. Далі ця стратегія передається у Context через конструктор або спеціальний метод встановлення.
    Коли Context повинен виконати дію, він не обирає алгоритм самостійно. Він просто викликає метод спільного інтерфейсу Strategy. Фактичну роботу виконує об’єкт ConcreteStrategy. Після завершення алгоритм повертає результат, а Context передає його далі у програму.
    Такий підхід дозволяє легко змінювати поведінку об’єкта без створення складної ієрархії наслідування та без дублювання коду.
Переваги використання патерну Strategy
    Основною перевагою патерну Strategy є зниження залежності між класами. Context не залежить від конкретних алгоритмів, а працює тільки через абстрактний інтерфейс. Це робить систему гнучкішою і полегшує її розвиток.
    Ще одна важлива перевага полягає у спрощенні додавання нової поведінки. Якщо потрібно реалізувати новий спосіб доставки або новий тип знижки, достатньо створити новий клас стратегії. Основний клас, який використовує алгоритм, можна залишити без змін.
    Також патерн покращує тестування. Оскільки кожна стратегія є окремим класом, її можна перевіряти незалежно від інших частин системи. Це особливо корисно у випадках, коли різні алгоритми мають складні правила або залежать від різних умов.
Недоліки та практичні сценарії використання
    Незважаючи на переваги, Strategy має і певні недоліки. Головний з них полягає у збільшенні кількості класів. Для кожного алгоритму створюється окрема реалізація, тому структура проєкту може стати більшою. Якщо в програмі є лише один простий алгоритм, використання патерну може бути зайвим ускладненням.
    Крім того, клієнтський код повинен знати, яку саме стратегію потрібно обрати. У складних системах для цього іноді створюють додаткові фабрики або конфігураційні механізми.
    На практиці Strategy доречно застосовувати в системах електронної комерції для розрахунку доставки, знижок або комісій, у фінансових сервісах для вибору правил обчислення, у графічних редакторах для різних інструментів обробки зображення, а також у програмах, де алгоритм має змінюватися залежно від налаштувань користувача.
4 ВИСНОВКИ
    У ході виконання практичної роботи було розглянуто патерн проєктування Strategy. Було визначено, що цей патерн належить до поведінкових патернів і використовується для інкапсуляції взаємозамінних алгоритмів в окремих класах.
    Було проаналізовано структуру патерну, зокрема ролі Strategy, ConcreteStrategy, Context та Client. Встановлено, що основна перевага патерну полягає у відокремленні алгоритму від об’єкта, який його використовує. Це дозволяє змінювати поведінку програми без внесення змін у головну бізнес-логіку.
    Також було розглянуто приклад реалізації мовою C#, у якому патерн Strategy застосовано для розрахунку вартості доставки. Приклад показав, що винесення алгоритмів в окремі класи робить код більш зрозумілим, розширюваним і зручним для тестування.
    У результаті виконання роботи було сформовано розуміння того, що Strategy є ефективним рішенням для систем, у яких існує декілька способів виконання однієї операції, а вибір конкретного алгоритму може змінюватися під час роботи програми.
